Agent 产品架构工程化深度落地指南

从「概率性幻觉」到「确定性交付」：一套面向复杂业务流程的智能体设计方法论。

2026-03 15 min read Archie

01架构演进的底层逻辑

在 Agent 开发的初期，开发者往往会陷入「全能智能体」的迷思。通过多次迭代发现，纯 Agent 架构在处理固定业务流时存在不可逾越的稳定性红线。

纯 Agent 架构
  • 逻辑漂移：Agent 随口答应无法实现的跳转
  • 幻觉节点：跳过必填表单直接执行结果
  • 状态丢失：Context 长度限制导致业务进度重置
分层治理架构
  • 确定性控制：代码控制状态机，Agent 仅执行节点任务
  • 数据隔离：Agent 只有建议权，写入逻辑归系统
  • 容错性：Agent 崩溃不影响业务状态持久化
核心架构哲学：决策权收归系统（代码控制 Flow），交互权交给 Agent（语义理解 Perception）。不要让 Agent 思考「下一步去哪」，而要告诉它「你现在就在 A 节点，请完成 A 的任务」。

02「铁轨与赛车」架构详解

针对多表单、多固定节点的任务，推荐采用外部状态机 + 局部 Agent 映射的架构。这类似于在铁轨上跑赛车——方向由轨道定，速度和推力由 Agent 提供。

表单处理的「草稿-确认」模式

当涉及表单填写节点时，Agent 的职责颗粒度应限制在：

  1. Slot Filling：从对话中提取 {"name": "张三", "date": "2026-03-08"}。
  2. 预填 (Pre-fill)：将提取的数据注入前端 UI 状态，用户在界面上可见。
  3. 人工确认 (HITL)：只有当用户点击「提交」按钮，系统才真正执行数据写入。
  4. 校验反馈：如果后端返回 error: 余额不足，系统将错误原文传给 Agent，由 Agent 翻译成「人话」引导用户修改。

03工程化落地的 6 大阶段

Phase 1

业务建模：划清「确定性」红线

不要试图用 Prompt 描述所有业务。画出 SOP 流程图，明确哪些节点是 Stateless（无状态咨询），哪些是 Transactional（事务操作）。为事务操作定义严格的 JSON Schema。

Phase 2

架构设计：建立「无状态」执行员

确保 Agent 每次对话不依赖自身的 Context 记忆，而是依赖系统注入的 Current_Global_State。将 Agent 拆解为原子化工具（Atom Tools），每个工具只负责一个极窄的 API 调用。

Phase 3

Prompt 协议化：解耦职责

拒绝「上帝 Prompt」。采用 Micro-Prompting 策略，为每个表单节点配置独立的 Prompt 模板。通过 Few-shot 样本强制 Agent 遵守该节点的「对话边界」。

Phase 4

护栏设计：防范逃逸

工程细节：使用模型原生支持的 Structured Output (Strict Mode)。在代码层设置拦截器，如果 Agent 试图跳转至非法状态，直接熔断并触发重试。
Phase 5

评估测试：黄金数据集

告别「体感测试」。建立由 100+ 极端 Case 组成的 Golden Dataset。引入 LLM-as-a-Judge（如使用 GPT-4o 评测业务模型），量化各节点的召回率与准确率。

Phase 6

观测迭代：链路追踪 (Tracing)

部署 LangSmith 或 Phoenix 进行全链路追踪。当 Bug 发生时，PM 需通过 Trace 分析：是模型理解偏了？还是 Tool Call 的参数传错了？还是后端业务报错了？

04Multi-Agent 的组织与协同规范

当单智能体承载信息过多时，必须将其拆分为「专家组」。

// 建议的 Agent 间通信协议 (Structured Communication)
{
  "sender": "Orchestrator",
  "receiver": "Form_Expert",
  "action": "REQUEST_FILL",
  "payload": {
    "form_id": "002",
    "missing_fields": ["phone_number"],
    "context": "User just mentioned their phone is 138..."
  },
  "max_retries": 2
}

多智能体协同的三大防线：

  • 1
    信令机制：Agent 之间不进行自由闲聊，只发送结构化信令。
  • 2
    防推诿逻辑：显式定义 Fallback_Agent，当两个 Worker Agent 逻辑冲突时，强制上报给 Orchestrator。
  • 3
    死循环阻断：在 Orchestrator 记录消息传递深度，超过 5 轮协同未达成结果则强制输出当前最优解或转人工。

05PM 的新维度：AI Native 系统设计

作为 AI PM，你需要完成从「页面设计者」到「系统调度者」的转变。

系统思维

理解状态机、解耦逻辑、数据流转闭环。

边界意识

识别 LLM 的局限性，设计优雅的 Fail-safe 机制。

数据工程

设计评测体系，从 Bad Case 中提取产品迭代规律。
