Agent 入门：新变化与新认知

基于 Google GEAR 团队演讲整理：Agent 的定义、核心组件、MCP 协议与三种主流架构模式。

2026-03 8 min read Aja Hammerly & Jason Davenport · 整理 by Archie

01什么是 Agent？

核心定义

Agent 是一个可以观察世界、使用工具与世界交互，从而完成目标的系统。它可以代表你行动，甚至自主行动来完成或发现目标。

值得注意：这个定义在过去 18 个月里一直保持稳定和一致——说明 Agent 的本质概念是持久的，不是昙花一现的 hype。

03三大核心组件

🧠
模型 (Model)
Agent 的「大脑」，负责推理和智能决策
🛠️
工具 (Tools)
与世界交互的手段，执行具体操作
📚
记忆 (Memory)
积累知识，让 Agent 越来越智能

模型：Thinking Models

现代 Agent 大多使用 「思考型模型」(Thinking Models)——这类模型内置了自我反思能力，能够在复杂任务中：

  • 考虑多个方案后再做出最终决策
  • 在复杂场景下评估不同响应路径
  • 对用户或其他 Agent 提供更智能的回复

工具：MCP 协议

Model Context Protocol (MCP)

MCP 类似于 API，但提供了额外的使用说明——包括 API 的使用方式、输入输出的含义。这让 Agent 能够理解何时、何地使用特定工具，而不只是机械地调用接口。

记忆与上下文管理

记忆不只是上下文窗口，还包括：

  • 短期记忆——当前会话的临时信息
  • 长期记忆——跨会话的持久知识
关键点：上下文需要主动管理——随着任务演进，可能需要压缩、摘要或丢弃不相关内容，或引入更多相关记忆。上下文管理的质量直接影响 Agent 的执行稳定性。

04三种主流架构模式

Pattern 01 简单模式 (Single Agent)

最基础的 Agent 架构：一个模型 + 指令 + 工具，没有循环或其他 Agent 参与。结构清晰，易于调试。

适用场景：目标明确、指令清晰的任务，或作为学习 Agent 行为的起点
Pattern 02 子 Agent 模式 (Sub-Agent)

主 Agent 处理大部分任务，将特定专业化任务委托给专门的子 Agent。通常会选择性地传递上下文。

适用场景：发票处理流程——文档提取 Agent 专门处理发票信息，主 Agent 负责整体流程推进
Pattern 03 编排器模式 (Orchestrator)

起始 Agent 作为路由器，将请求转发给最合适的专业 Agent。识别用户意图是关键挑战，需要快速且准确。

适用场景：电商前台 Agent——根据订单、商品、支持等不同意图，路由到相应的专业 Agent
核心要点
  • Agent = 观察能力 + 工具能力 + 目标导向
  • 三大组件：模型（大脑）、工具（手脚）、记忆（知识库）
  • Thinking Models 具备自我反思能力，适合复杂决策
  • MCP 协议让工具调用更智能，不只是 API 调用
  • 上下文需要主动管理：压缩、摘要、清理、补充
  • 三种主流架构：简单模式 → 子 Agent → 编排器，复杂度递增
